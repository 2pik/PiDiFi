# Вклад в проект PDFToPPTXConverter

Этот документ описывает процесс внесения изменений в проект.

## Как внести свой вклад

### 1. Сообщение об ошибке

Если вы нашли ошибку, создайте issue с подробным описанием:
- Версия macOS
- Версия Python
- Шаги для воспроизведения ошибки
- Логи приложения (из ~/Library/Logs/PDFToPPTXConverter/)

### 2. Предложение новой функции

Перед созданием pull request:
1. Создайте issue с описанием предлагаемой функции
2. Дождитесь обсуждения и подтверждения
3. Приступайте к реализации

### 3. Pull Request

Процесс создания PR:

1. **Форкните репозиторий**
2. **Создайте ветку** для вашей функции или исправления:
   ```bash
   git checkout -b feature/your-feature-name
   # или
   git checkout -b fix/issue-number-description
   ```

3. **Внесите изменения**, следуя принципам:
   - Код должен быть чистым и понятным
   - Избегайте дублирования
   - Добавляйте комментарии только там, где это необходимо
   - Следуйте стилю существующего кода

4. **Запустите тесты**:
   ```bash
   make test
   ```

5. **Проверьте код линтером**:
   ```bash
   make lint
   ```

6. **Закоммитьте изменения** с понятными сообщениями:
   ```bash
   git commit -m "fix: исправлена ошибка конвертации изображений"
   # или
   git commit -m "feat: добавлена поддержка пакетной обработки"
   ```

7. **Отправьте в удалённый репозиторий**:
   ```bash
   git push origin feature/your-feature-name
   ```

8. **Создайте Pull Request** на GitHub

## Стандарты кода

### Стиль кода

- Используйте UTF-8 кодировку
- Максимальная длина строки: 120 символов
- Отступы: 4 пробела
- Имена переменных: snake_case
- Имена классов: PascalCase
- Константы: UPPER_CASE

### Документация

- Все публичные функции должны иметь docstring
- Docstring должен описывать назначение, параметры и возвращаемое значение
- Сложная логика должна быть прокомментирована

### Тестирование

- Новые функции должны сопровождаться тестами
- Минимальное покрытие для критических функций: 80%

## Процесс ревью

Все pull request проходят обязательное ревью:
1. Проверка соответствия стандартам кода
2. Проверка наличия тестов
3. Функциональное тестирование
4. Одобрение мейнтейнера

## Лицензия

Отправляя код, вы соглашаетесь с тем, что он будет распространяться под лицензией MIT.
